# ADR-0029: ai-guide の Reference 軸を維持型から生成型へ移行する

- **Status**: accepted
- **Date**: 2026-06-13
- **Related**: [`doc-strategy/index.md`](../doc-strategy/index.md) (改訂対象の配置規約) / [ADR-0008](0008-mutate-revert-replacement.md) (移管先の更新節を持つ ADR)

## コンテキスト

ai-guide 4 軸のうち Reference 軸 (`code-map.md` + in-tree README) は維持型 — 実装変更のたびに
手で同期する運用だった。Phase 2 リファクタリングの事前調査で code-map 全 24 節をコード現物と
突き合わせたところ、8 節に stale が見つかり、その**全てがコード結合度の高い記述**（ファイル一覧・
生成箇所パス・enum 値・実行環境種別・目次アンカー）に集中していた。一方、散文で書かれた判断・
知見はほぼ無傷で、同時に監査した ADR 本文も 24 本中 23 本が現物と一致していた。
この先 10 本超の構造変更 PR（flat layout / feature-first 再構成等）が控えており、維持型のままでは
drift 面積がさらに拡大する。Reference 軸の保存戦略を決める必要があった。

## 選択肢

- **A. 維持型の現状化を継続**: stale 8 節を修正し、以後も実装 PR ごとに code-map / README を手動同期する。
- **B. 生成型へ移行**: 非導出情報を恒久の家へ移管して code-map を削除し、README / アプローチ仕様書は
  skill で必要時に生成する（生成物はスナップショットとして常設コミット）。

ここで**非導出情報** = コードから再構成できない情報（設計判断・研究上の主張・既知の限界・実測値・
運用上の落とし穴）、**導出情報** = コードが一次出典でそこから再構成できる情報（モジュール構成・
ファイル責務・データフロー・契約の転記）と定義する。

### 評価

| | A 維持型継続 | B 生成型移行 |
|---|---|---|
| drift 再発リスク | × 構造変更のたび同期漏れ（今回 8/24 節で実証） | ○ 保存しない情報は陳腐化しない |
| 情報喪失リスク | ○ 何も動かさない | ○ 全 24 節の行き先を仕分けで事前確認（補足） |
| 読み手の取得コスト | ○ 常に即読 | △ スナップショットは即読、最新化には生成実行が必要 |
| 維持コスト | × 全構造変更 PR に同期作業が乗る | ○ 生成時のみ（検証パス込み） |

## 決定

**B（生成型へ移行）** を採用する。具体的には:

1. **`code-map.md` を廃止する**。非導出情報は種類ごとの恒久の家へ移管する:
   - 設計判断 → `adr/`（既存 ADR の更新節への追記を含む）
   - 研究上の主張・既知の限界 → `current-research.md`（主張側 / §妥当性の脅威）
   - 実測値 → `research/<approach>/notes/`（実行コミット・コマンド・日付の印を必須とする）
   - 局所的な運用上の落とし穴 → 該当コードの現地コメント

   導出情報（モジュール構成・ファイル責務・データフロー）は**保存せず、必要時に生成する**。
2. **in-tree README とアプローチ仕様書は skill 生成物とする**。読込対象は文書種別で分ける:
   in-tree README は該当コード + ADR 索引・関連 ADR のみを根拠にし、current-research /
   research notes は読み込まない (README は流動的な上位文書を参照しない規約のため)。
   アプローチ仕様書はこれに加えて current-research の該当節 + research notes を読み込む。
   いずれも file:line 引用付きで生成する。ヘッダに生成時コミットハッシュと「生成物・手編集禁止」を
   印字し、スナップショットとして常設コミットする。生成後に引用箇所を現物と突き合わせる
   検証パスを必須とする。
3. **実測値は二段運用とする**:
   - **論文モード**: `research/<approach>/code/` の再走スクリプトを、最新契約との整合確認込みで
     再実行した値のみを使う。
   - **理解モード**: `research/<approach>/notes/` の保存値を使ってよいが、コミット印の併記を必須とする。
4. **`adr/README.md` の索引は「論文の主張から ADR を逆引きする」用途に寄せる**。対象領域列は
   ファイル単位でなくモジュール粒度に丸め、drift 面積を縮小する。

主要な根拠:

- drift はコード結合度の高い記述に限局していた（stale 8 節はすべてパス・数値・一覧の類で、散文の
  判断・知見は無傷）。「非導出情報だけを保存し、導出情報は保存しない」という切り分けが実態と一致する。
- 生成の正確性は調査ワークフロー自体が実証した: コードを読んで記述を再構成する生成で約 95% が
  現物一致し、残る誤りは引用突き合わせの検証パスで捕捉できた。
- code-map 全 24 節の行き先を事前に仕分けし（補足の表）、削除で失われる情報がないことを確認済み。

## 結果 / 影響

採用によって得られるもの:

- Reference 軸の drift が構造的に消える（保存しない情報は陳腐化しない）。構造変更 PR から
  「code-map 同期」の作業が消える。
- 1 軸 1 出典の徹底: 判断は ADR、主張・限界は current-research、実測は research notes、
  構成・責務はコード自身（+ 生成スナップショット）が一次出典になる。
- README の鮮度がコミット印で機械的に判定できる（「いつ時点の記述か」が常に明示される）。

諦めるもの・将来のコスト:

- 「全体を 1 ファイルで通読する」onboarding 導線。代替はアプローチ仕様書の必要時生成。
- 文書としての git 履歴の連続性: 生成物は世代ごとに差し替わるため、blame で文章の経緯を追えない
  （判断の経緯は ADR 側が担う）。
- 最新の README が必要なときは再生成の実行コスト（skill 実行 + 検証パス）を毎回払う。
- 生成物への手編集は次回生成で消えるため、「手編集禁止」の規律が必要。

`doc-strategy/index.md` の Reference 軸の記述（4 軸表の code-map 行・in-tree README 節・
参照方向規約）は本 ADR の内容で改訂する。

## トリガー (再検討の条件)

以下の条件のいずれかが成立したらこの ADR を見直す:

- 生成物の誤りが検証パスをすり抜ける事例が反復し、生成の正確性が担保できないと判明した。
- コミット印が古いまま放置された生成物に起因する誤解・手戻りが実際に発生した（再生成の運用が
  回っていない兆候）。
- 上記 4 種の恒久の家のどれにも収まらない非導出情報が繰り返し発生し、受け皿として維持型の
  Reference 文書が再び必要になった。

トリガー発火時は新しい ADR を起票し、本 ADR は `superseded by ADR-NNNN` に書き換える。
