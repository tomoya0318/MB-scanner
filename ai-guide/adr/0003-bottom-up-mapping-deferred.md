# ADR-0003: bottom-up mapping の実装を第 2 段階以降に遅延する

- **Status**: accepted
- **Date**: 2026-04-23
- **Related**: PR #8, [ADR-0002](0002-babel-topdown-subtree-hash.md)

## コンテキスト

GumTree の AST diff アルゴリズム (Falleri+ ASE 2014) は 2 段階:

1. **top-down phase**: 大きな isomorphic subtree をハッシュ一致で検出
2. **bottom-up phase**: top-down で未マッチのノードを対象に、子の一致率・形状類似度でマッチング (主にリネーム対応)

本 PR では ADR-0002 の通り top-down phase を自作する。bottom-up を同時に実装するか、後回しにするかが判断事項。

Selakovic データセットの性質:

- 大半の変形パターンは **同じ patch / コミット由来** であり、変数名・関数名の変更 (リネーム) は稀
- 「共通ノード」判定は名前の厳密一致で取れるケースがほとんど
- C1〜C4 条件 (第 2 段階) のうち C3 (エイリアス) だけはリネーム追跡が効く可能性がある

## 選択肢

- **A. PR #1 で bottom-up も同時実装**: GumTree 完全実装に近づける
- **B. top-down のみでスタートし、必要性を実測で判断してから追加**: 第 2 段階着手時にデータで判断

### 評価

| 軸 | A (同時実装) | B (遅延) |
|---|---|---|
| 追加実装コスト | +1 日、~250 行 | 0 (当面) |
| 実測前の判断 | 投機的投資 | データドリブン |
| Selakovic での効果 | 不明 (推定では低) | 第 2 段階で実測してから |
| スコープ膨張 | 大きい (PR #1 が重くなる) | 小さい |

## 決定

**B (遅延)** を採用。主要な根拠:

- Selakovic はリネーム稀なので top-down で当面足りる公算が高い
- 第 2 段階 C1 / C2 / C4 は基本リーフマッチングで足りる
- C3 (エイリアス) で初めて必要になる可能性があり、そこで実測判断が妥当
- PR #1 のスコープを膨らませないことが優先

## 結果 / 影響

**得るもの:**
- PR #1 が top-down の基盤構築に専念できる
- 「必要なら +1 日で追加できる」柔軟性

**諦めるもの:**
- リネーム変形を含むペアでは偽陰性 (common ノードを diff と判定) が起きうる

## トリガー (bottom-up 実装条件)

以下のいずれかが成立したら bottom-up mapping を追加する:

- 第 2 段階の C3 (エイリアス条件) で、リネーム検出が明確に必要と実測
- Selakovic 10 パターン integration テスト (PR #4) で top-down だけでは取りこぼすケースが出た
- OSS 適用フェーズで偽陰性率が許容できないレベルと判明

実装時は `FastSubtreeSet` クラスに `mapTo(node: t.Node): t.Node | null` を追加する前提で API 互換を保つ。

## 補足

論文での位置づけは「GumTree top-down phase のみを Babel AST 上で実装。bottom-up はデータセット特性 (リネーム変形が稀) から本研究のスコープでは未採用」と明記する。
