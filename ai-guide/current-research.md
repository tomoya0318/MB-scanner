# 研究計画 — 指導教員確認用

---

## 背景

### 未解決問題: (slow, fast) ペアから「条件付き検出パターン」を自動導出する

「遅い書き方と速い書き方のペアから，**どこでその改善が使えるか**を自動検出する」ためには，**構造と条件の両方**が必要:

| 側面 | 問い | 例 (Pattern #1: for..in → Object.keys) |
|---|---|---|
| **構造** | どの AST ノードがパターンに必須か | `for..in` + `hasOwnProperty` ガードが必須．変数名や本体は自由 |
| **条件** | どんな条件で安全に適用できるか | 対象が Object 型 + prototype が汚染されていない |

**構造だけでは FP だらけ．条件だけでは検出対象がない．両方揃って初めて使える検出器になる．**

### Hydra の解法: SMT による記号実行

**Hydra (OOPSLA 2024)** は LLVM IR で**構造と条件の両方を N=1 で自動導出**した．SMT ソルバ (論理式の成立を全入力について自動判定するツール) を使い，各 AST ノードを抽象化しても等価性が保たれるかを**全入力に対して**検証する (pruning)．

Hydra が可能だった理由: **LLVM IR の整数演算は SMT の決定可能な断片に収まる**ため，全入力を記号的に網羅できる．

### JavaScript で同じことができない理由

JS の動的意味論 (動的型付け，prototype chain，closure，`eval` 等) は SMT で完全にモデル化できず，**記号実行のパスが無限に爆発する**．

ただし JS 向けの記号実行ツール (ExpoSE 等) は存在し，JS の**部分的な断片** (単純な算術，文字列操作等) は記号的に扱える．問題はパスが有限に収まらないこと．

### SYDIT の教訓: 構造だけでは不十分

**SYDIT (FSE 2011)** は N=1 で構文パターンの導出を試みたが，**条件を付けなかった**ため FP が多すぎた．**LASE (ICSE 2013)** で同著者が「N=1 は精度不足」と批判し，以降は複数例前提の研究 (Getafix OOPSLA 2019, REVISAR ICSE 2018, PyCraft FSE 2024) に流れた．

### Selakovic の手動前提条件

**Selakovic & Pradel (ICSE 2016)** は 16 の JS プロジェクトから 98 件の修正済みパフォーマンス issue を収集し，10 の再帰パターンに **5 種類の前提条件** (T/NF/P/TF/V) を手動で付与した．§7.2 で「**自動チェックは困難であり今後の課題**」と結論．

データセット: https://github.com/marijaselakovic/JavaScriptIssuesStudy

### 10 年間誰も取り組んでいない (調査済み)

被引用 114 件の全件確認 + 広域検索の結果，JS で構造の自動導出も条件の自動チェックも行った後続研究は **0 件**．

## 課題

JS で「条件付き検出パターンの自動導出」を実現するには:

1. **構造の導出**: どの AST ノードがパターンに必須かを判定
2. **条件の特定**: どんな条件で安全に適用できるかを判定
3. **条件による記号実行パスの有限化**: JS 全体の記号実行は無理だが，条件で入力空間を分割すればパスを有限にできる可能性がある

Hydra は 1〜3 を SMT で一括解決した (LLVM IR だから可能)．JS ではこの 3 つを**段階的に**解く必要がある．

---

## 目的

> **(slow, fast) ペアから条件付き検出パターンを自動導出する手法を段階的に確立する．**

## 3 段階のアプローチ

本研究は以下の 3 段階で問題を解く．**本論文で 3 段階すべてを実装・評価する．**

```
第 1 段階: 構造パターンの自動導出
  (slow, fast) ペアから実行ベースの pruning で構文パターンを導出

第 2 段階: 条件による入力空間の有限分割
  C1〜C4 条件軸で入力空間を有限クラスに分割し，
  各クラスの代表値で等価性を検証

第 3 段階: C1〜C4 による体系的検証 (同値分割テスト)
  構造パターン + C1〜C4 条件を組み合わせ，
  条件ごとの有限クラスで等価性が保たれるかを体系的に検証
```

### 3 段階の関係

| 段階 | 前提 | 実現すること | 出力 |
|---|---|---|---|
| **第 1 段階** | なし | 構造パターン導出 | ワイルドカード付き AST パターン |
| **第 2 段階** | 第 1 段階のパターン | 条件軸で入力空間を有限分割 | パターン + C1〜C4 条件 |
| **第 3 段階** | 第 1 + 第 2 段階 | 各条件クラスの代表値で体系的検証 | パターン + 条件 + 各クラスの検証結果 |

各段階は前の段階を前提とする:
- 第 2 段階はパターンがないと条件を付ける対象がない
- 第 3 段階は条件がないと入力空間を有限に切れない

## アプローチ

### パイプライン

```
入力: (slow, fast) ペア + セットアップコード

  [1] Pruning → 構造パターン導出 (第 1 段階)
  [2] C1〜C4 条件抽出 + 同値分割テスト (第 2〜3 段階)
  [3] ts-eslint 検出ルール生成

出力: 条件付き検出ルール (Level 別の確信度付き)

※ 等価性検証器 (vm + 4 オラクル) は [1] と [2] の内部で使う共通基盤
```

### 等価性検証器: 4 オラクル (O1〜O4)

`slow ≡ fast` を判定する基盤ツール．Node.js の `vm` で隔離実行し，以下 4 軸で出力を観測する．

| # | オラクル | 観測内容 | 必要な理由 (Selakovic パターン) |
|---|---|---|---|
| **O1** | 戻り値 | 式の評価結果 (deep equal) | for..in の列挙結果，`String()` の文字列化結果 |
| **O2** | 引数変異 | 入力オブジェクトの実行前後の構造差分 | Array 操作 (push/splice) 等の破壊的メソッドの影響 |
| **O3** | 例外 | throw の有無・型・メッセージ | prototype 汚染で `hasOwnProperty` が壊れた時の TypeError |
| **O4** | 外部観測 | console / グローバル変数 / I/O の差分 | DOM/I/O 系パターン (Selakovic では少数) |

イテレーション順序・Promise 解決値は O1 に集約 (配列に集める / `await` 後の値を比較)．時間計算量は等価性の対象外．

#### C1〜C4 (条件) と O1〜O4 (オラクル) の関係

C と O は直交する別概念．**C は入力空間を分割し，O は出力を観測する．**

| | C1〜C4 | O1〜O4 |
|---|---|---|
| 役割 | 入力空間の分割 | 出力の観測 |
| 問い | どんな入力で検証するか | 何を見て等価と判定するか |
| 出現段階 | 第 2〜3 段階 | 全段階の検証器内部 |

検証器の内部で両者は掛け合わさる:

```
for 各 C 軸のクラス:
  代表値 v を選ぶ
  result_slow = run(slow, v)   ← O1〜O4 を観測
  result_fast = run(fast, v)   ← O1〜O4 を観測
  if any O_i で result_slow ≠ result_fast → そのクラスで非等価
```

<details>
<summary><strong>具体例: Pattern #1 の C2 形状軸 × オラクル</strong></summary>

| C2 クラス | 代表値 | O1 戻り値 | O3 例外 |
|---|---|---|---|
| clean proto | `{a:1}` | 等価 | 等価 |
| 汚染 proto | `Object.create({x:1})` | **非等価** (slow が `x` も列挙) | 等価 |
| `hasOwnProperty` 上書き | `{hasOwnProperty:()=>true}` | 等価 | **非等価** (slow が throw) |

→ C2 軸を有限クラスに切り (C)，各クラスで何が壊れるかを 4 オラクルで観測 (O) する．

</details>

### 第 1 段階: 実行ベース Hydra 式 pruning

Hydra の SMT ベース pruning を，JS では**実行ベースの等価性検証器**で代替する．さらに **AST 差分フィルタ**を前段に置き，単一セットアップでの誤 prune を構造的に防ぐ．

**AST 差分フィルタ** (Babel AST 上で GumTree top-down phase [Falleri+ 2014] を実装):
- slow のノードのうち、fast に同型サブツリーが存在しないもの = **差分ノード**
- 差分ノードは「fast に対応物がない = パターンの本質」として pruning 候補から除外 (試行せず必須扱い)
- 共通ノード (fast にも同型存在) のみを pruning 候補にする

**pruning 本体**:

```
差分フィルタ通過後の候補ノード N について:
  - N をリネーム ($VAR) または削除して slow' を作る
  - 等価性検証器で slow' ≡ fast を setup 上で確認
  - 等価 → N は不要 → ワイルドカード化
  - 非等価 → N は必須 → 残す
→ 必須ノードだけ残った最小構文パターンが出力
```

初回の検証 (pruning 前の slow ≡ fast) で等価性が確認できなければ，そのペアは不正として除外される．**事前の等価性検証ステップは pruning に吸収される**．

<details>
<summary><strong>具体例: Pattern #1 (for..in → Object.keys)</strong></summary>

```js
// slow (元)
for (var key in obj) {
  if (obj.hasOwnProperty(key)) { use(key); }
}
```

| ノード | 差分フィルタ判定 | pruning 試行 | 判定 |
|---|---|---|---|
| `var` | 共通 (候補入り) | ワイルドカード化可 | **不要** → `$DECL` |
| `key` | 共通 (候補入り) | リネーム可 | **不要** → `$ITER` |
| `obj` | 共通 (候補入り) | リネーム可 | **不要** → `$TARGET` |
| `use(key)` | 共通 (候補入り) | 削除/置換可 | **不要** → `$BODY` |
| `hasOwnProperty` チェック | **差分 (fast に非存在)** | 試行せず | **必須** |

自動導出結果:
```
for ($DECL $ITER in $TARGET) {
  if ($TARGET.hasOwnProperty($ITER)) { $BODY }
}
```

</details>

### Hydra (SMT) との違い

| | Hydra | 本研究 |
|---|---|---|
| 対象言語 | LLVM IR | JavaScript |
| 検証手段 | SMT (**全入力を保証**) | 実行ベース (**有限クラスの代表値で検証**) |
| AST 差分解析 | なし (SMT が一括解決) | Babel AST + GumTree top-down phase [Falleri+ 2014] |
| 出力 | 構造 + 条件 (SMT が一括解決) | 構造 (第 1 段階) + 条件 (第 2 段階) + 体系的検証 (第 3 段階) |

**Unsoundness の緩和**:
- 第 1 段階の AST 差分フィルタで「fast に対応物がない差分ノード」を必須として保護 (単一セットアップでの誤 prune を構造的に回避)
- 共通ノードに潜む edge case (差分フィルタで救えないケース) は第 3 段階 C1〜C4 同値分割テストで代表値網羅性を確保する二段構造
- pruning 候補の静的除外は **`@babel/types` の文法メタデータから自動導出する効率最適化** (ADR-0005) であり、正確性は Babel 型検査 / round-trip parse / 等価性検証器による多層防御で担保する (実装詳細は [`code-map.md` §Pruning エンジン](code-map.md#pruning-エンジン))

### 第 2 段階: C1〜C4 条件抽出

第 1 段階で導出したパターンに条件を付与する．C1〜C4 は Selakovic の 5 前提条件 (T/NF/P/TF/V) を再構成したもの:

| Selakovic | 本研究 | 対応 |
|---|---|---|
| T (Type check) | **C1 型** | 引き継ぎ |
| NF + P + TF | **C2 形状** | 統合 |
| (なし) | **C3 エイリアス** | **新規追加** |
| V (Value check) | **C4 値の出自** | 引き継ぎ + provenance 分類 |

<details>
<summary><strong>各軸の確認手段</strong></summary>

| 軸 | TS での確認手段 | JS でも使える確認手段 |
|---|---|---|
| **C1 型** | 明示的型注釈 + TS 推論 | `--checkJs` 推論 / `typeof` ガード |
| **C2 形状** | TS interface + AST | AST 走査 (TS/JS 共通) |
| **C3 エイリアス** | TS 型情報 | ESLint scope manager (TS/JS 共通) |
| **C4 値の出自** | 型 + ガード | リテラル値 / 制御フロー内ガード |

</details>

### 第 3 段階: C1〜C4 同値分割テスト

C1〜C4 の各軸で入力空間を有限クラスに分割し，各クラスの代表値で等価性を検証する．**記号実行が全入力を網羅するのに対し，本手法は有限個の代表値で近似する**．

| 軸 | クラス分割の例 (Pattern #1) | 代表値 |
|---|---|---|
| C1 型 | Object / Array / Map / null | `{a:1}`, `[1,2]`, `new Map()`, `null` |
| C2 形状 | clean proto / 汚染 proto / 上書き | `{a:1}`, `Object.create({x:1})`, `{hasOwnProperty:()=>true}` |
| C3 alias | なし / あり | `const obj = {a:1}`, `const ref = obj` |
| C4 値 | 通常 / Symbol / 空 | `{a:1}`, `{[Symbol()]:1}`, `{}` |

各クラスの代表値で等価性検証器を実行し，**どの条件クラスで等価性が壊れるか**を記録 → ablation の結果が自動で得られる．

#### Graded Soundness (段階的確信度)

検出時に C1〜C4 のどこまで確認できたかで確信度を付与:

| Level | 確認内容 | ESLint での扱い |
|---|---|---|
| 0 | パターン不一致 | 出力しない |
| 1 | 構文パターン一致 (第 1 段階) | suggestion |
| 2 | + C1 型確認 | info |
| 3 | + C2 形状確認 | warning |
| 4 | + C3 エイリアスなし | error |
| 5 | + C4 値の出自確認 | error (高確信) |

## 難しさ

### 本研究固有の困難

| 困難 | 内容 |
|---|---|
| **SMT が使えない** | JS の動的意味論は SMT の決定可能断片に落ちない．実行ベースに転換する必要がある |
| **Unsound pruning** | 実行ベースではテスト入力のみ保証．AST 差分フィルタで差分ノードを必須保護し、共通ノードの edge case は第 3 段階 C1〜C4 で緩和 |
| **セットアップの代表性** | テスト入力が入力空間を十分に網羅しているか保証できない |

### 妥当性の脅威

| 脅威 | 緩和策 |
|---|---|
| 等価性検証器をパイプラインと評価の両方で共用 | ネガティブセットアップの一部を手動確認 |
| セットアップの設計偏り | OSS 適用で新たな失敗パターンを発見する可能性を議論 |
| Selakovic データの経年 (2016 年) | OSS ケーススタディで 2026 年時点の有効性を確認 |
| 検証器が一部の candidate を「中身まで実行できず判定不能 (`inconclusive`)」にする (dep 不在 / Ember の AMD loader / workload が薄い等) | `equal` = positive な等価エビデンスで確認・`not_equal` = 差を観測・`inconclusive` = 検証不能、と verdict を分け、「検証器が著者判断と一致した」の分母は `equal`+`not_equal` の確認済み分のみにし `inconclusive` は別途「検証カバレッジ」として報告する。`not_equal` のうち記録 Proxy 干渉等の checker artifact 由来のものは手動レビューで角に置く (= ADR-0018 / ADR-0013 §threats)。当面 ~67 件確認済 / ~6 件 not_equal (うち 4 件が genuine な検出、うち 1 件 = angular-10351 は著者判断との既知の不一致) / ~33 件 inconclusive |

### パターン限定の記号実行 (sound 検証への拡張)

本論文の第 3 段階は同値分割テスト (代表値による具体的検証) だが，構造パターン + C1〜C4 条件でパターン内の実行パスが有限に限定されるため，**ExpoSE 等の JS 記号実行ツールで sound に検証できる可能性**がある:

- 構造パターンが制御フローを固定 → ループ回数・分岐が限定
- C1〜C4 が入力の型・形状・値を限定 → 各分岐のパスが削減
- **「JS 全体は記号実行不能だが，パターン + 条件で限定された断片なら可能」**
